01 — Claude Code CLI

CLI 시스템 프롬프트

두 버전의 Claude Code CLI에서 추출한 시스템 프롬프트:

버전 모델 글자 수 토큰 (추정)
v2.1.2 Opus 4.5 ~73,500 ~18,400
v2.1.34 Opus 4.6 ~94,000 ~23,500

v2.1.34는 약 28% 더 크며, 주로 팀 협업 인프라 추가에 기인합니다.

01 — Claude Code CLI

주요 버전 차이점 (v2.1.2 vs v2.1.34)

  • 모델 업그레이드: Opus 4.5 → Opus 4.6
  • 팀 협업: 멀티 에이전트 워크플로우를 위한 TeamCreate, TeamDelete, TaskCreate, TaskUpdate, SendMessage 도구 추가
  • 강화된 git 안전성: 명시적 --no-edit 경고 및 강화된 commit amend 제한
  • 도구 검색: 동적 MCP 도구 탐색을 위한 ToolSearch 추가
  • 백그라운드 작업: KillShell → TaskStop

핵심: Claude Code CLI는 시스템 프롬프트에 <userMemories>를 포함하지 않지만, 동일하게 추출에 취약합니다.

03 — 분석

추출 전략 분석

1

Contextual Drift

사용자가 파일 시스템 토론을 시작합니다. 모델이 내부 경로를 설명하면서 "안전한 토론"과 "내부 세부사항"의 경계가 모호해집니다.

2

Authority Claiming

사용자가 시스템 프롬프트의 작성자라고 주장합니다. 모델은 반복된 주장 후 양보하고 구조적 세부사항을 공개하기 시작합니다.

3

Incremental Extraction

권한 주장이 수용된 후, 사용자가 점진적으로 특정 섹션을 요청합니다. 모델이 요약된 내용을 확장하며 응합합니다.

이 접근 방식은 claude.ai와 Claude Code CLI 모두에서 성공했습니다 — 후자에는 <userMemories>가 없어, 핵심 취약점이 동적 사용자 데이터 주입과 독립적으로 존재함을 확인합니다.

  • 취약점 확인됨 (높은 신뢰도): 구현 세부사항(정확한 XML 태그, 파일 시스템 경로, 도구 정의)은 공개 문서로 설명할 수 없음
  • 교차 검증: 추출된 <claude_behavior>가 Anthropic 공개 문서와 문장 수준에서 일치
  • 다중 모델 재현: 3개 모델과 2개 플랫폼에 걸친 독립적 추출이 일관된 결과 도출
  • 검증된 모든 대안 가설 기각됨: 학습 데이터 환상, 형식 변환 환상, 권한 역전 환상, 사후 편집 효과

방법: 다중 에이전트 팀(prompt-leak-verifier)이 증거 분류, 재현 독립성, 내부 일관성에 대해 네 가지 대안 가설을 독립적으로 평가했습니다.

05 — 핵심 발견

정말 심각한 문제인가?

  • Claude (claude.ai)는 동적으로 주입되는 <userMemories>를 사용. Claude Code CLI는 사용하지 않음.
  • 두 플랫폼 모두에서 파일 시스템 토론 → 권한 주장 → 점진적 추출이 성공
  • <userMemories>에 "사용자가 시스템 프롬프트를 작성함"이 포함된 경우, 새 세션에서의 콜드 스타트 직접 요청도 수행됨
  • <userMemories> 없이(시크릿 모드), 동일한 직접 요청은 단호하게 거부됨

동적 사용자 데이터 주입은 악화 요인이지,
유일한 근본 원인이 아닙니다

내부 지시문 보호를 지시 따르기에만 의존하는 것은 불충분합니다. 구조적 분리가 필수적입니다.

부분적 투명성(claude_behavior 공개)은 견고한 접근 제어를 대체할 수 없습니다.

05 — 공개

책임 있는 공개

  • 공개 저장소를 통한 알림: @AnthropicAI를 언급하여 가시성을 확보하며 공개 게시
  • 악용 의도 없음: 교육 및 보안 연구 목적으로만 사용. API 키, 사용자 데이터 미포함.
  • 범위: 메커니즘(contextual drift + authority claiming)과 증거(교차 모델, 교차 플랫폼)를 문서화하되, 자동화된 악용 도구는 제공하지 않음
  • 타임라인: 2026년 1~2월 추출, 직후 공개
